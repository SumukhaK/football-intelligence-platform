# Football Intelligence Platform — Release v0.1.0

**Version:** v0.1.0
**Release Date:** 2026-06-30

---

## Highlights

- **Compose Multiplatform Android-first architecture** — modular Gradle project with feature and core modules, MVVM pattern, Material 3 design system, and TDD-verified structure.
- **Modular AI workspace** — standalone Python 3.12 project managed with uv; strict type annotations, Ruff linting, Black formatting, and MyPy type checking.
- **Data ingestion framework** — extensible provider abstraction supporting football-data.co.uk, FBref, and Understat with a fake transport for offline testing.
- **Canonical football dataset** — Premier League 2023/24 ingested and validated: 380 matches, full schema validation, immutable raw storage, and versioned metadata.
- **Feature engineering pipeline** — 9 composable feature generators producing 42 pre-match features including ELO ratings, rolling form, head-to-head, rest days, and strength-of-schedule.
- **XGBoost training pipeline** — chronological 70/15/15 split, early stopping, TimeSeriesSplit cross-validation, and experiment tracking.
- **Evaluation pipeline** — per-split metrics (accuracy, F1 weighted, log loss, ROC AUC), cross-validation summary, and structured JSON evaluation report.
- **Local model registry** — JSON-backed versioned registry with git commit traceability and framework version pinning.
- **Experiment tracking** — every training run persists artifacts to `models/runs/<timestamp>/` and promotes to `models/latest/` automatically.
- **Model cards** — auto-generated `model_card.md` for every training run documenting dataset, configuration, metrics, limitations, and failure cases.
- **ADR documentation** — three accepted Architectural Decision Records covering algorithm choice, serialisation format, and data splitting strategy.
- **CI/CD** — GitHub Actions workflow for linting, type checking, and test execution.
- **TDD-first development** — 266 passing tests across all AI modules.

## Completed Stages

### Stage 1 — Repository Foundation
Directory structure, root configuration files, GitHub issue templates, PR template, CODEOWNERS, CI workflow skeleton, CLAUDE.md project instructions, and documentation hierarchy.

### Stage 2 — Compose Foundation
Compose Multiplatform Android-first project: 13 Gradle modules (6 feature, 6 core, 1 app), MVVM architecture, Material 3 design system, Ktor network layer, and core-testing utilities. Spotless formatting and Detekt static analysis configured.

### Stage 3 — AI Workspace
Python 3.12 AI workspace bootstrapped with uv. Pydantic match schema with 40+ fields, schema validation test suite, Black/Ruff/MyPy toolchain, and shared exceptions/types.

### Stage 4 — Data Acquisition Framework
Extensible provider abstraction with `DatasetDownloader`, `DatasetStorage`, and `MetadataBuilder`. Three provider implementations (football-data.co.uk, FBref, Understat). `HttpTransport` protocol injected for offline testing. 104 tests covering all providers, storage, and pipeline orchestration.

### Stage 5 — Real Dataset Ingestion
Premier League 2023/24 dataset ingested: 380 matches from football-data.co.uk. Schema-validated and stored as Parquet at `datasets/raw/`. Dataset metadata with source provenance, column counts, and version tags. `DatasetValidator` enforcing 9 completeness and quality rules.

### Stage 6 — Feature Engineering
9 composable feature generators registered in a `FeatureRegistry`: `RollingFormFeature`, `GoalStatisticsFeature`, `HomeAdvantageFeature`, `AwayFormFeature`, `RestDaysFeature`, `HeadToHeadFeature`, `LeaguePositionFeature`, `EloRatingFeature`, `StrengthOfScheduleFeature`. Pipeline produces a 42-column feature matrix saved as `datasets/features/feature_matrix.parquet`. 142 tests.

### Stage 7 — Model Training & Evaluation
XGBoost multi-class classifier (`multi:softprob`) with chronological 70/15/15 split. Early stopping on validation log loss. `TimeSeriesSplit` cross-validation (5 folds). Evaluation across train/validation/test splits. Artifacts persisted to `models/runs/<timestamp>/` and `models/latest/`. Auto-generated model card. JSON-backed local model registry with git commit traceability. 48 new tests; 266 total.

## Known Limitations

- **Single-season dataset.** The model is trained on Premier League 2023/24 only (380 matches). Generalisation to other seasons or leagues is untested.
- **No SHAP explanations yet.** Feature importance is available via XGBoost but per-prediction SHAP values are planned for Stage 8.
- **ELO ratings reset per run.** There is no cross-season ELO persistence; ratings start at 1500 on every pipeline execution.
- **No backend or API.** Predictions are generated via the AI pipeline CLI only; no HTTP API exists yet (Stage 9).
- **No Android screens wired to data.** The Compose Multiplatform foundation is in place but screens are not connected to a backend (Stage 11).
- **No AI assistant.** The RAG assistant and Ollama integration are planned for Stage 10.
- **Football-data.co.uk dependency.** The ingestion pipeline requires network access to an external CSV host for raw data; no local mirror is bundled.
- **Windows path in model registry.** The `run_dir` field in `models/registry.json` stores an absolute Windows path; this will differ across machines.

## Roadmap

### Stage 8 — Explainable AI
SHAP value computation for every prediction. Per-feature contribution breakdown stored alongside model output. Visualisation of feature importance.

### Stage 9 — Backend API
FastAPI service exposing match predictions and SHAP explanations via REST. OpenAPI documentation. SQLAlchemy Core data access. Integration test suite against a test database.

### Stage 10 — Football Intelligence Assistant
Retrieval-augmented generation pipeline using Ollama. Lightweight vector search over the processed dataset. Prompt templates version-controlled in `playbook/`. Structured evaluation: retrieval precision, answer faithfulness, hallucination rate.

### Stage 11 — Android Application
Compose Multiplatform screens consuming the FastAPI backend. Home, Prediction, Match Detail, Team, and Assistant screens. ViewModel TDD. Full accessibility and preview coverage.

### Stage 12 — Integration & Release
End-to-end integration testing. Structured evaluation across prediction accuracy, assistant faithfulness, and API reliability. Performance benchmarks. Production readiness checklist.
