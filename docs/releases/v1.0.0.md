# Football Intelligence Platform — Release v1.0.0

**Version:** v1.0.0
**Release Date:** 2026-07-01

---

## Highlights

- **Complete Android application** — 8-screen Compose Multiplatform app (Home, Match Prediction, Prediction Result, Explain Prediction, AI Assistant Chat, Model Information, Settings, About) consuming all five backend endpoints via Ktor.
- **MVVM with TDD** — ViewModels backed by `StateFlow<UiState>`, repositories returning `NetworkResult<T>`, 9 repository unit tests written test-first.
- **Koin dependency injection** — modular DI with `AndroidX Navigation Compose` routing across 8 screens.
- **End-to-end integration test suite** — 36 integration tests in `ai/tests/integration/` covering all API endpoints with real model artifacts, error recovery, and latency validation.
- **Performance benchmarks** — prediction latency ~3 ms, SHAP explanation ~7 ms, health check ~5 ms, all well within thresholds.
- **Production readiness validated** — all 462 tests passing, all quality gates clean, Android build successful, every API endpoint verified against real model output.
- **Complete documentation** — 9 stage reports, 8 demo scripts, CLI reference, quick-start guide, troubleshooting guide, 4 ADRs, and this release document.

## Completed Stages

### Stage 1 — Repository Foundation
Directory structure, root files, GitHub templates (bug report, task, ADR request), PR template, CODEOWNERS, CI workflow skeleton, CLAUDE.md project instructions, and documentation hierarchy.

### Stage 2 — Compose Multiplatform Foundation
Compose Multiplatform Android-first project: 13 Gradle modules (6 feature, 6 core, 1 app), MVVM architecture, Material 3 design system, Ktor network layer, and core-testing utilities. Spotless and Detekt configured.

### Stage 3 — AI Workspace
Python 3.12 AI workspace bootstrapped with uv. Pydantic match schema with 40+ fields, schema validation test suite, Black/Ruff/MyPy toolchain, and shared exceptions/types.

### Stage 4 — Data Acquisition Framework
Extensible provider abstraction: `DatasetDownloader`, `DatasetStorage`, `MetadataBuilder`. Three provider implementations (football-data.co.uk, FBref, Understat). Injectable `HttpTransport` for offline testing. 104 tests.

### Stage 5 — Canonical Dataset
Premier League 2023/24 ingested: 380 matches from football-data.co.uk. `DatasetValidator` enforcing 9 quality rules. Separate `RawMatch` and `ProcessedMatch` schemas. Versioned raw/processed Parquet storage.

### Stage 6 — Feature Engineering
9 composable feature generators in a `FeatureRegistry` with Kahn's topological sort. `.shift(1)` on all rolling features for leakage prevention. 42-column feature matrix saved as `datasets/features/feature_matrix.parquet`. 142 tests.

### Stage 7 — Model Training
XGBoost `multi:softprob` classifier with chronological 70/15/15 split and early stopping. `TimeSeriesSplit` cross-validation (5 folds). Per-run artifact persistence, auto-generated model card, JSON model registry with git commit traceability. 48 new tests; 266 total.

### Stage 8 — Explainable AI
`SHAPExplainer` wrapping `shap.TreeExplainer` with multi-class output normalisation. `ExplainerCache` to avoid rebuilding per request. `LocalExplanation` and `GlobalSummary` Pydantic models. Generated artifacts: JSON explanations, beeswarm, waterfall, force, and dependence plots. ADR 004 accepted. 54 new tests; 320 total.

### Stage 9 — FastAPI Backend
Five REST endpoints exposing Stage 7 and Stage 8 AI capabilities plus the assistant. Lifespan-based dependency injection. Structured exception handling (503/422/500). Request logging middleware. OpenAPI schema at `/docs` and `/redoc`. 43 backend tests; 363 total.

### Stage 10 — Football Intelligence Assistant
Local RAG pipeline: `DocumentLoader` → `TextChunker` → `OllamaEmbedder` → numpy `VectorStore` → cosine retrieval → system prompt → `OllamaGenerator`. `POST /assistant/chat` backend endpoint with graceful 503 when Ollama is unavailable. All Ollama calls mocked in tests. 52 new tests; 426 total.

### Stage 11 — Compose Multiplatform Android Application
8-screen Android application built with Compose Multiplatform (KMP) and connected to all five backend endpoints via Ktor. Screens: Home (health status), Match Prediction, Prediction Result, Explain Prediction (SHAP), AI Assistant Chat, Model Information, Settings, About. MVVM with StateFlow-backed ViewModels, Koin DI, AndroidX Navigation Compose routing, FootballTheme with Material 3.

### Stage 12 — End-to-End Integration & Production Readiness
36 integration tests in `ai/tests/integration/` covering all 5 API endpoints, error recovery, prediction pipeline correctness, and explainability pipeline correctness. Performance benchmarks: prediction <100 ms, SHAP <2 s, health <50 ms. Full test suite: 462 passing (426 unit + 36 integration).

## Known Limitations

- **Ollama required for assistant.** The RAG assistant requires a locally running Ollama server with `nomic-embed-text` and `llama3.2` pulled. The backend starts gracefully without it but returns 503 on assistant requests.
- **Single-season dataset.** The model is trained on EPL 2023/24 only (380 matches). Cross-season generalisation is untested.
- **ELO ratings reset per run.** No cross-season Elo persistence; ratings start at 1500 on every pipeline execution.
- **Windows absolute path in model registry.** The `run_dir` field in `models/registry.json` stores the local machine's absolute path.
- **No authentication or rate limiting.** The FastAPI backend has no auth, no API keys, and no request throttling. Not suitable for public deployment as-is.
- **Android neutral features for demo.** `buildNeutralFeatures()` provides 44 neutral values for demo predictions. Real feature computation from match context is a future enhancement.
- **Football-data.co.uk dependency.** Raw data ingestion requires network access to an external CSV host.
- **Vector store not in Git.** The RAG index must be rebuilt with `ollama pull nomic-embed-text` and `uv run python -m assistant.pipeline --rebuild`.

## Roadmap

No further stages are planned within this project scope. Potential future enhancements:

- Multi-season dataset ingestion with cross-season Elo persistence.
- Hyperparameter optimisation with Optuna or similar.
- Structured RAG faithfulness evaluation with a ground-truth Q&A set.
- Android feature computation from real match context (replacing neutral demo features).
- Authentication and rate limiting for deployment beyond local development.
- PostgreSQL backend for production deployment.

## Running the Platform

### AI Pipeline (from `ai/`)

```sh
# 1. Install dependencies
uv sync --extra dev

# 2. Ingest data
uv run python -m scripts.ingest_football_data

# 3. Engineer features
uv run python -m feature_engineering.pipeline

# 4. Train model
uv run python -m training.pipeline

# 5. Generate SHAP explanations
uv run python -m explainability.pipeline

# 6. Start the backend
uv run uvicorn backend.app.main:app --reload
```

### AI Assistant (requires Ollama)

```sh
ollama pull nomic-embed-text
ollama pull llama3.2

cd ai
uv run python -m assistant.pipeline --rebuild
uv run uvicorn backend.app.main:app --reload
```

### Android (from `frontend/`)

```sh
./gradlew assembleDebug          # macOS/Linux
.\gradlew.bat assembleDebug      # Windows PowerShell
```

### Integration Tests

```sh
# From ai/ — requires backend running with real model artifacts
uv run pytest tests/integration/ -v
```
