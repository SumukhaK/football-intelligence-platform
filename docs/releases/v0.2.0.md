# Football Intelligence Platform — Release v0.2.0

**Version:** v0.2.0
**Release Date:** 2026-06-30

---

## Highlights

- **Complete AI data pipeline** — end-to-end flow from raw CSV download through feature engineering, model training, and SHAP explainability, all CLI-runnable in under 15 seconds.
- **Feature engineering pipeline** — 9 composable generators producing 42 pre-match features with leakage prevention via `.shift(1)` and Kahn's topological dependency resolution.
- **XGBoost prediction model** — chronological 70/15/15 split, early stopping, TimeSeriesSplit cross-validation, 56.1% test accuracy on Premier League 2023/24.
- **SHAP explainability pipeline** — `shap.TreeExplainer` over the XGBoost booster; per-prediction local explanations and global feature importance summaries with beeswarm, waterfall, force, and dependence plots.
- **Local model registry** — JSON-backed versioned registry with git commit traceability, framework version pinning, and per-run artifact persistence.
- **Experiment tracking** — every training run persists to `models/runs/<timestamp>/`; `models/latest/` always reflects the current best.
- **FastAPI backend** — four REST endpoints (`/health`, `/model`, `/predict`, `/explain`), pydantic-settings config, lifespan dependency injection, structured error handling, and auto-generated OpenAPI documentation.
- **Prediction API** — `POST /predict` accepts match features and returns predicted outcome (H/D/A) with per-class probabilities and confidence.
- **Explainability API** — `POST /explain` returns the full SHAP feature contribution breakdown alongside the prediction.
- **Football Intelligence Assistant** — local RAG pipeline using Ollama for both embeddings and generation; answers questions grounded in the platform's own knowledge base.
- **Retrieval-Augmented Generation** — `DocumentLoader` → `TextChunker` → `OllamaEmbedder` → numpy `VectorStore` → cosine retrieval → `OllamaGenerator` pipeline.
- **Ollama integration** — configurable chat model (default: `llama3.2`) and embedding model (default: `nomic-embed-text`); all model names are environment-variable overridable.
- **Local vector database** — numpy-backed vector store persisted to disk; no external vector DB required.
- **Grounded responses with citations** — system prompt enforces source-only answering; every response includes `[source: <filename>]` citations; low-relevance chunks filtered at retrieval time.
- **Full CI pipeline** — GitHub Actions running ruff, black, mypy, and pytest on every push and PR.
- **Architecture documentation** — 4 accepted ADRs, 7 stage reports, CLI reference, quick-start guide, troubleshooting guide, and 6 demo scripts for technical interviews.

## Completed Stages

### Stage 1 — Repository Foundation
Directory structure, root files, GitHub templates (bug report, task, ADR request), PR template, CODEOWNERS, CI workflow skeleton, CLAUDE.md project instructions, and documentation hierarchy.

### Stage 2 — Compose Multiplatform Foundation
Compose Multiplatform Android-first project: 13 Gradle modules (6 feature, 6 core, 1 app), MVVM architecture, Material 3 design system, Ktor network layer, and core-testing utilities. Spotless formatting and Detekt static analysis configured.

### Stage 3 — AI Workspace
Python 3.12 AI workspace bootstrapped with uv. Pydantic match schema with 40+ fields, schema validation test suite, Black/Ruff/MyPy toolchain, and shared exceptions/types.

### Stage 4 — Data Acquisition Framework
Extensible provider abstraction: `DatasetDownloader`, `DatasetStorage`, `MetadataBuilder`. Three provider implementations (football-data.co.uk, FBref, Understat). `HttpTransport` protocol injected for offline testing. 104 tests.

### Stage 5 — Canonical Dataset
Premier League 2023/24 ingested: 380 matches from football-data.co.uk. `DatasetValidator` enforcing 9 quality rules. Separate `RawMatch` and `ProcessedMatch` schemas. Versioned raw/processed Parquet storage.

### Stage 6 — Feature Engineering
9 composable feature generators in a `FeatureRegistry` with Kahn's topological sort. `.shift(1)` on all rolling features for leakage prevention. 42-column feature matrix saved as `datasets/features/feature_matrix.parquet`. 142 tests.

### Stage 7 — Model Training
XGBoost `multi:softprob` classifier with chronological 70/15/15 split and early stopping. `TimeSeriesSplit` cross-validation (5 folds). Per-run artifact persistence, auto-generated model card, JSON model registry with git commit traceability. 48 new tests; 266 total.

### Stage 8 — Explainable AI
`SHAPExplainer` wrapping `shap.TreeExplainer` with multi-class output normalisation. `ExplainerCache` to avoid rebuilding per request. `LocalExplanation` and `GlobalSummary` Pydantic models. Generated artifacts: JSON explanations, beeswarm, waterfall, force, and dependence plots. ADR 004 accepted. 54 new tests; 320 total.

### Stage 9 — FastAPI Backend
Four REST endpoints exposing Stage 7 and Stage 8 AI capabilities. Lifespan-based dependency injection. Structured exception handling (503/422/500). Request logging middleware. OpenAPI schema at `/docs` and `/redoc`. 43 backend tests; 363 total.

### Stage 10 — Football Intelligence Assistant
Local RAG pipeline: `DocumentLoader` → `TextChunker` → `OllamaEmbedder` → numpy `VectorStore` → cosine retrieval → system prompt → `OllamaGenerator`. `POST /assistant/chat` backend endpoint with graceful 503 when Ollama is unavailable. All Ollama calls mocked in tests. 52 new tests; 426 total.

## Known Limitations

- **No Android screens connected to data.** The Compose Multiplatform foundation is complete but UI screens remain unimplemented placeholders (Stage 11).
- **Ollama required for assistant.** The RAG assistant requires a locally running Ollama server with `nomic-embed-text` and `llama3.2` pulled. The backend starts gracefully without it but returns 503 on assistant requests.
- **Single-season dataset.** The model is trained on EPL 2023/24 only (380 matches). Cross-season generalisation is untested.
- **ELO ratings reset per run.** No cross-season Elo persistence; ratings start at 1500 on every pipeline execution.
- **Windows absolute path in model registry.** The `run_dir` field in `models/registry.json` stores the local machine's absolute path.
- **No authentication or rate limiting.** The FastAPI backend has no auth, no API keys, and no request throttling. Not suitable for public deployment.
- **No end-to-end integration tests.** Unit and API tests pass; an end-to-end test suite spanning all components is planned for Stage 12.
- **Football-data.co.uk dependency.** Raw data ingestion requires network access to an external CSV host.

## Running the Platform

### AI Pipeline (from `ai/`)

```sh
# 1. Install dependencies
uv sync --extra dev

# 2. Ingest data
uv run python -m scripts.ingest_football_data

# 3. Engineer features
uv run python -m feature_engineering.pipeline

# 4. Train model
uv run python -m training.pipeline

# 5. Generate SHAP explanations
uv run python -m explainability.pipeline

# 6. Start the backend
uv run uvicorn backend.app.main:app --reload
```

### AI Assistant (requires Ollama)

```sh
# Pull required models
ollama pull nomic-embed-text
ollama pull llama3.2

# Build the knowledge index
cd ai
uv run python -m assistant.pipeline --rebuild

# The backend will load the index automatically on startup
uv run uvicorn backend.app.main:app --reload
```

### Android (from `frontend/`)

```sh
./gradlew assembleDebug
```
