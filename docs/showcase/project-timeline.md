# Project Timeline

A stage-by-stage history of how the Football Intelligence Platform was built, from an empty repository to a tested, documented, end-to-end product.

Each stage was implemented on its own feature branch, merged via pull request, and documented with a dedicated stage report and demo guide before the next stage began.

---

## Stage 1 — Repository Foundation

**Purpose:** Establish the repository skeleton, contribution standards, and CI scaffolding before any feature code is written.

**Outcome:** A repository any contributor can clone and immediately understand — directory layout, commit conventions, and review checklist all defined up front.

**Major Deliverables:**
- Top-level directory structure (`ai/`, `frontend/`, `backend/`, `docs/`, `datasets/`, `scripts/`, `tools/`)
- GitHub issue templates (bug report, task, ADR request), PR template, `CODEOWNERS`
- CI workflow skeleton (GitHub Actions)
- `.claude/CLAUDE.md` — project instructions and architecture rules
- Documentation hierarchy under `docs/`

**Git Tag:** —

## Stage 2 — Compose Multiplatform Foundation

**Purpose:** Stand up the Android project structure early so the eventual frontend has a correct module architecture from day one, rather than retrofitting one later.

**Outcome:** 13 Gradle modules with MVVM conventions, Material 3 theming, and a Ktor network layer scaffold — ready to receive feature implementation in Stage 11.

**Major Deliverables:**
- 13-module Gradle project (6 feature modules, 6 core modules, 1 app module)
- MVVM architecture conventions documented
- Material 3 design system scaffold
- Ktor network layer foundation
- `core-testing` utilities module
- Spotless formatting + Detekt static analysis configured

## Stage 3 — AI Workspace

**Purpose:** Bootstrap the Python data engineering and ML workspace with the same engineering rigour as the Android side — typed, linted, tested from the first commit.

**Outcome:** A `uv`-managed Python 3.12 workspace with a 40+ field Pydantic match schema and the full Black/Ruff/MyPy toolchain wired into CI.

**Major Deliverables:**
- `ai/` workspace bootstrapped with `uv`
- Pydantic `Match` schema (40+ fields)
- Schema validation test suite
- Black/Ruff/MyPy toolchain configuration
- Shared exceptions and types module

**Git Tag:** —

## Stage 4 — Data Acquisition Framework

**Purpose:** Build a provider-agnostic data ingestion abstraction before committing to a single data source, so additional providers can be added without rewriting the ingestion layer.

**Outcome:** An extensible `DatasetDownloader` / `DatasetStorage` / `MetadataBuilder` framework with three working provider implementations, fully testable offline via an injected `HttpTransport`.

**Major Deliverables:**
- `DatasetDownloader`, `DatasetStorage`, `MetadataBuilder` abstractions
- Three provider implementations: football-data.co.uk, FBref, Understat
- `HttpTransport` protocol for offline test injection
- 104 tests

**Git Tag:** —

---

## Stage 5 — Canonical Dataset

**Purpose:** Ingest a real, validated dataset rather than continuing to develop against synthetic fixtures.

**Outcome:** 380 Premier League 2023/24 matches ingested, validated against 9 quality rules, and persisted as versioned Parquet — the canonical dataset every later stage builds on.

**Major Deliverables:**
- Premier League 2023/24 dataset ingested (380 matches)
- `DatasetValidator` enforcing 9 quality rules
- Separate `RawMatch` and `ProcessedMatch` schemas
- Versioned raw/processed Parquet storage

**Git Tag:** —

---

## Stage 6 — Feature Engineering

**Purpose:** Convert the canonical match dataset into a model-ready feature matrix, with explicit leakage prevention since this is fundamentally a time-series problem.

**Outcome:** 42 pre-match features generated by 9 composable generators, dependency-ordered via topological sort, with `.shift(1)` applied to every rolling-window feature to guarantee no future-information leakage.

**Major Deliverables:**
- 9 composable feature generators (`FeatureRegistry`)
- Kahn's topological sort for generator dependency ordering
- `.shift(1)` leakage prevention on all rolling features
- 42-column feature matrix (`datasets/features/feature_matrix.parquet`)
- 142 tests

**Git Tag:** —

---

## Stage 7 — Model Training

**Purpose:** Train the XGBoost prediction model with a methodology appropriate for time-series sports data, not a naive random split.

**Outcome:** A trained `multi:softprob` XGBoost classifier with chronological 70/15/15 split, `TimeSeriesSplit` cross-validation, and a versioned model registry tracing every artifact back to its producing git commit.

**Major Deliverables:**
- XGBoost `multi:softprob` classifier with early stopping
- Chronological 70/15/15 train/validation/test split
- `TimeSeriesSplit` cross-validation (5 folds)
- Per-run artifact persistence and auto-generated model card
- JSON model registry with git commit traceability
- 48 new tests (266 total)
- [ADR 001](../adr/001-use-xgboost-for-predictions.md), [ADR 002](../adr/002-joblib-model-serialization.md), [ADR 003](../adr/003-chronological-train-val-test-split.md) accepted

**Git Tag:** v0.1.0

---

## Stage 8 — Explainable AI

**Purpose:** Make every prediction explainable as a first-class capability, not a post-hoc analysis script.

**Outcome:** A `SHAPExplainer` wrapping `shap.TreeExplainer`, with per-version caching and generated global/local explanation artifacts (beeswarm, waterfall, force, dependence plots).

**Major Deliverables:**
- `SHAPExplainer`, `ExplainerCache`, `ExplainabilityPipeline`, `ExplanationService`
- Multi-class SHAP output normalisation
- `LocalExplanation` and `GlobalSummary` Pydantic models
- Generated artifacts: JSON explanations, beeswarm, waterfall, force, dependence plots
- [ADR 004](../adr/004-shap-for-explainability.md) accepted
- 54 new tests (320 total)

**Git Tag:** —

---

## Stage 9 — FastAPI Backend

**Purpose:** Expose the prediction and explainability pipelines through a real HTTP API with production-shaped error handling, not a script invoked manually.

**Outcome:** A FastAPI service with 4 REST endpoints (`/health`, `/model`, `/predict`, `/explain`), lifespan-based dependency injection, structured exception handling, and auto-generated OpenAPI docs.

**Major Deliverables:**
- FastAPI application with `/health`, `/model`, `/predict`, `/explain`
- `pydantic-settings` configuration
- Lifespan-based dependency injection (no global state)
- Structured exception handling (503/422/500)
- Request logging middleware
- OpenAPI schema at `/docs` and `/redoc`
- 43 backend tests (363 total)

**Git Tag:** —

---

## Stage 10 — Football Intelligence Assistant

**Purpose:** Add a natural-language interface to the platform's data that is grounded — answers must come from retrieved data, never from the model's parametric knowledge alone.

**Outcome:** A local RAG pipeline (Ollama embeddings + numpy vector store + Ollama generation) exposed via `POST /assistant/chat`, with graceful 503 degradation when Ollama is unavailable.

**Major Deliverables:**
- `DocumentLoader` → `TextChunker` → `OllamaEmbedder` → numpy `VectorStore` → retriever → `OllamaGenerator` pipeline
- `POST /assistant/chat` endpoint
- Graceful 503 handling when Ollama is unreachable
- All Ollama calls mocked in the test suite
- 52 new tests (426 total)

**Git Tag:** v0.2.0

---

## Stage 11 — Compose Multiplatform Android Application

**Purpose:** Build the real Android client against the real backend — not a UI mockup, but a working consumer of all five API endpoints.

**Outcome:** An 8-screen Compose Multiplatform app with MVVM architecture, Ktor networking, Koin DI, and AndroidX Navigation Compose, fully wired to the live FastAPI backend.

**Major Deliverables:**
- 8 screens: Home, Match Prediction, Prediction Result, Explain Prediction, AI Assistant Chat, Model Information, Settings, About
- MVVM with `StateFlow`-backed ViewModels (`androidMain`) and stateless Composables (`commonMain`)
- Ktor client covering all 5 backend endpoints
- Koin dependency injection, AndroidX Navigation Compose routing
- `FootballTheme` — Material 3 light/dark colour scheme
- 9 repository unit tests (JUnit 5 + MockK)

**Git Tag:** —

---

## Stage 12 — End-to-End Integration & Production Readiness

**Purpose:** Validate that every component built across Stages 1–11 actually works together — against the real trained model, not mocks — and that the system fails gracefully under every tested failure mode.

**Outcome:** 36 integration tests against real model artifacts, measured performance benchmarks well within target thresholds, and a documented production-readiness assessment.

**Major Deliverables:**
- 36 integration tests: prediction pipeline, explainability pipeline, all 5 API endpoints, error recovery
- Performance benchmarks: prediction ~3 ms, SHAP explanation ~7 ms (both far under threshold)
- Full test suite: 462 passing (426 unit + 36 integration)
- `docs/demo/stage-12-demo.md` — complete end-to-end demo guide
- `docs/reports/stage-12-summary.md` — production readiness assessment

**Git Tag:** v1.0.0
